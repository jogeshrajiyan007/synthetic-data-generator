# Data Quality Monitoring – List of DQ Rules

**Last Refreshed:** February 26, 2026

---

## 📊 What This Page Tells You

This page is the **rule registry** for the entire data quality framework. It gives you full transparency into every validation check applied to incoming HR data — what is being checked, against which field, under which category, and what happens when a record fails. Use this page to understand the logic behind DQ scores, validate that rules are correctly configured, and investigate why specific errors are being raised.

---

## 🔢 KPI Definitions

| KPI | What it means |
|-----|--------------|
| **Job Data Rules Applied** | Number of DQ rules actively evaluating **Job Data (JD)** fields — the largest rule set at 182 rules, covering fields like assignment category, career events, FTE, business unit, etc. |
| **PI Data Rules Applied** | Number of rules evaluating **Personal Information (PI)** fields such as address, date of birth, and city — 43 rules in total. |
| **CP Data Rules Applied** | Number of rules evaluating **Compensation (CP)** fields — 2 rules currently active. |
| **Total DQ Rules Applied** | The combined count of all active rules across all data categories — **227 rules** in total running every week against incoming records. |

## 📋 Rule Table — Column Definitions

Each row in the table represents one individual DQ rule. Here is what each column means:

| Column | What it means |
|--------|--------------|
| **Rule ID** | A sequential identifier for the rule within the dashboard. Used for reference when investigating errors. |
| **Functional ID** | The unique business code for the rule (e.g. `DQ-VA-0001`, `DQ-CO-0016`). The prefix indicates the DQ dimension: **VA** = Validity, **CO** = Completeness. Use this ID to trace a rule back to its source definition or raise it in a support ticket. |
| **DQ Dimension** | Which quality dimension this rule enforces — **Validity** (correct format or allowed value), **Completeness** (field must be filled), or **Consistency** (logical alignment between fields). |
| **Datapoint** | The specific HR field being validated by this rule (e.g. Address, FTE, Assignment Category, Career Events). |
| **File Category** | The source file type the rule applies to — **JD** (Job Data), **PI** (Personal Information), or **CP** (Compensation). |
| **Data Category** | The data category classification of the field being checked — mirrors the File Category in most cases. |
| **Log Type** | The system that logs the error when this rule fails — **MDM** (Master Data Management system) or **HDL** (HCM Data Loader). This tells you where to look when investigating a failure. |
| **Error Code** | The specific error or warning code raised when this rule is violated. Key codes include: **ERROR** (hard failure, record rejected), **ERROR_MAP01** (value not in allowed mapping list), **ERROR_MDY01** (mandatory field missing), **WARNING_LEN01** (format length violation — warning only, record may still pass). |
| **Description** | Plain-language explanation of exactly what the rule checks and what a valid record must look like. Always read this before raising a remediation request with a source entity. |

## 🧭 How to Navigate

- **Click "Data Collect – Interface Monitoring"** → Returns to the data collection and file volume view.
- **Click "DQ Monitoring"** → Goes to the DQ score dashboard to see how these rules are performing across entities.
- **Click "Investigate Errors"** → Opens the row-level error log where you can match failed records to specific Rule IDs and Functional IDs from this page.
- **Scroll the rule table** → The table lists all 227 rules in order. Use the Rule ID or Functional ID to locate a specific rule quickly.

---

> 💡 **Tip:** When an entity reports a data quality issue, start here — look up the **Functional ID** from the error log on the Investigate Errors page, find the matching row in this table, and read the **Description** column. This tells you exactly what the entity needs to fix in their source system before the next file submission.
