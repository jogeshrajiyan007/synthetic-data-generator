## 2. Data Modeler

> Responsible for designing conceptual, logical, and physical data models that accurately represent business entities and relationships.

---

### 🟦 DM-01 — Star Schema Design
```
Based on all uploaded datasets, design a complete Star Schema for an insurance data warehouse.
For each Fact Table provide:
- Table name and grain (what one row represents)
- All measurable facts/metrics with data types
- Foreign keys to dimension tables
- Recommended partitioning and clustering strategy

For each Dimension Table provide:
- Table name and description
- Natural key and surrogate key
- All descriptive attributes with data types
- SCD Type recommendation (SCD1 / SCD2 / SCD3) with justification
- Conformed dimensions shared across multiple fact tables

Also identify any degenerate dimensions or junk dimensions appropriate for the model.
```

---

### 🟦 DM-02 — Normalisation Assessment
```
Perform a complete normalisation assessment across all uploaded datasets.
For each dataset:
1. Identify the current normal form (UNF, 1NF, 2NF, 3NF, BCNF)
2. List all normalisation violations with specific column examples
3. Propose step-by-step normalisation to achieve 3NF
4. Identify all candidate keys, primary keys, and foreign key relationships
5. Flag any columns that represent repeating groups or multi-valued attributes

Provide before/after table structures for each normalisation step.
```

---

### 🟦 DM-03 — Entity Relationship Analysis
```
Identify all business entities across the uploaded datasets and define their relationships.
For each entity:
- Entity name and description
- Key attributes and identifiers
- Cardinality of relationships (1:1, 1:N, M:N)
- Whether the relationship is identifying or non-identifying
- Any associative entities needed for M:N resolution

Present findings as a written ER narrative and a list of all relationships in the format:
[Entity A] — [Cardinality] — [Entity B] : [Relationship description]
```

---

### 🟦 DM-04 — Source to Target Mapping (STTM)
```
Generate a complete Source-to-Target Mapping (STTM) for all uploaded datasets.
For every source column, produce a mapping row with:
| Source File | Source Column | Source Data Type | Transformation Logic | Target Table | Target Column | Target Data Type | Key Type | Nullable | Notes |

Transformation logic must specify one of:
- Direct mapping (no change)
- Type cast (from → to)
- Derived / calculated (formula)
- Lookup / reference join (join condition)
- Null handling strategy
- Standardisation rule (date format, casing, trim)

Cover every column across every uploaded file. Do not skip any column.
```

---

### 🟦 DM-05 — Data Vault Design Proposal
```
Based on the uploaded insurance datasets, propose a Data Vault 2.0 modelling approach as an alternative to the star schema.
Include:
1. Hub tables — list all business keys and their hub definitions
2. Link tables — all associations and transactions between hubs
3. Satellite tables — all descriptive context and history per hub/link
4. Reference tables — any code or lookup tables identified
5. Point-in-time (PIT) tables recommended for performance
Explain the business justification for choosing Data Vault over Kimball for each entity where applicable.
```

## 4. Data Quality Analyst

> Responsible for profiling, measuring, monitoring, and improving the quality of data assets.

---

### 🟨 DQA-01 — Full Data Quality Report
```
Perform a comprehensive data quality assessment across all uploaded datasets.
For each column in each dataset, evaluate and score the following DQ dimensions:
1. Completeness — % non-null values
2. Accuracy — values within expected domain/range
3. Consistency — same values represented the same way across datasets
4. Uniqueness — duplicate detection
5. Timeliness — data freshness indicators
6. Validity — format and constraint compliance
7. Integrity — referential integrity between related datasets

Present an overall DQ score per dataset (0–100) and a heatmap-style summary table.
Flag all critical violations that need immediate attention.
```

---

### 🟨 DQA-02 — Duplicate Record Analysis
```
Analyse all uploaded datasets for duplicate records.
For each dataset:
1. Identify exact duplicates (all columns identical)
2. Identify near-duplicates (same key fields, different metadata)
3. Identify potential master record conflicts across datasets (same entity in multiple files)
4. Quantify duplicates: count, percentage of total records
5. Propose deduplication strategy and recommended master record selection logic
6. Estimate business impact of unresolved duplicates (incorrect aggregations, inflated KPIs)
```

---

### 🟨 DQA-03 — Data Quality Rules Definition
```
Based on all uploaded datasets, define a complete set of data quality rules for a DQ monitoring framework.
For each rule specify:
- Rule ID
- Rule name
- Affected table and column
- Rule type (Not Null, Referential Integrity, Range Check, Format Check, Uniqueness, Business Rule)
- Rule expression / SQL logic
- Severity (Critical / High / Medium / Low)
- Expected pass rate threshold (e.g. 99.5%)
- Recommended action on failure

Output as a structured DQ rules catalogue ready for implementation in a DQ tool (e.g. Great Expectations, Soda, dbt tests).
```

---

### 🟨 DQA-04 — Anomaly & Outlier Detection Report
```
Analyse all numerical and categorical columns in the uploaded datasets for anomalies and outliers.
For numerical columns:
- Identify values beyond 3 standard deviations from the mean
- Identify impossible values (negative ages, future claim dates, etc.)
- Flag zero-value anomalies in monetary columns

For categorical columns:
- Identify unexpected or non-standard category values
- Detect inconsistent formatting (mixed case, abbreviations vs full names)
- Flag values that appear suspiciously infrequent (< 0.1% of records)

Provide recommended handling strategy for each anomaly type.
```

---

### 🟨 DQA-05 — Cross-Dataset Consistency Check
```
Perform a cross-dataset consistency analysis across all uploaded files.
Check for:
1. Referential integrity — do foreign key values in one dataset exist as primary keys in another?
2. Semantic consistency — same real-world entity named differently across datasets
3. Temporal consistency — dates that should align across datasets but don't
4. Aggregate consistency — summary totals in one file that don't reconcile with row-level data in another
5. Currency/amount consistency — same transaction amounts recorded differently

Report all inconsistencies with specific examples from the data.
```
