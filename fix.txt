Insurance Data Warehouse Star Schema Design Prompt

Analyze all uploaded insurance datasets and design a complete enterprise-grade Star Schema.

For each source dataset:

Identify the business process (Policy, Claims, Billing, Payments, Underwriting, Customer, Agent, Coverage, etc.)

Explain what business entity the dataset represents

Recommend whether it should become a Fact, Dimension, Bridge, or Reference table


For each Fact Table provide:

Table Name

Business Purpose (why the table exists in an insurance data warehouse)

Grain (what exactly one row represents)

Detailed Fact Table Description (insurance process captured, business events tracked, reporting and analytics use cases)

Measures/Metrics with:

Column Name

Data Type

Business Definition

Insurance Meaning

Calculation Logic (if derived)

Additive / Semi-Additive / Non-Additive classification


Foreign Keys and referenced dimensions

Example business questions answered

Recommended partitioning and clustering strategy with justification


For each Dimension Table provide:

Table Name

Business Purpose

Detailed Dimension Description (entity represented and its role in insurance reporting)

Natural Key and Surrogate Key

Attributes with:

Column Name

Data Type

Business Meaning

Example Value

Reporting Usage


Recommended SCD Type (1/2/3) with business justification

Hierarchies (if applicable)

Example business questions answered


Additional Requirements

Identify conformed dimensions shared across fact tables.

Highlight insurance KPIs enabled by the model (Written Premium, Earned Premium, Loss Ratio, Claim Frequency, Claim Severity, Retention Rate, etc.).

Explain all assumptions made from the source data.

Present the final output in a professional data-modeling document format suitable for architecture review.


Important: Do not provide generic table names or brief descriptions. Every fact, dimension, measure, and attribute must include insurance-domain context, business purpose, and reporting significance.
