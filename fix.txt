# Investigate Errors – Data Collect Rows with DQ Errors

**Last Refreshed:** February 26, 2026 &nbsp;|&nbsp; **Coverage:** 08 Feb 2026 – 14 Feb 2026

---

## 📊 What This Page Tells You

This is the **deepest level of investigation** in the dashboard. It exposes every individual employee record that was rejected due to a data quality rule failure — organised by date, entity, and employee ID — alongside the exact field values that caused the failure. Use this page when you need to go beyond scores and percentages and identify the specific rows and fields that need to be corrected at source.

---

## 🔢 KPI Definitions

| KPI | What it means |
|-----|--------------|
| **Employees Number** | Total employee records processed this week across all entities. |
| **Files Loaded** | Number of source files ingested from Oracle PaaS this week. |
| **Overall Error Rate** | Percentage of all loaded records containing at least one DQ rule violation. At 64.79%, over half of all records have at least one issue this week. |
| **Total DQ Rules Applied** | The total number of active validation rules (227) that were run against incoming records this week. |
| **Rejected Rows** | Total number of records that failed one or more DQ rules and were excluded from downstream processing — 2,276 rows this week. |

## 📋 Visual Breakdown

### Employee Rows Rejected Due to DQ Rule Fails (Matrix Table)
The core visual of this page — a **hierarchical matrix** that drills from date → entity → employee ID → file name, with each row's field values displayed as columns across the top (ADDR1, ADDR2, ADDR3, ADDR4, ALTERNATE_CHAR_NAME, ANNUAL_PERCENT_WORKED, ASSIGNMENT_CATEGORY, ASSIGNMENT_STATUS, and more).

The hierarchy works as follows:
- **Level 1 — Reported Date** (e.g. Monday, February 09, 2026): Groups all rejections by the date the file was processed.
- **Level 2 — Entity** (e.g. AXA Belgium): Shows which AXA entity submitted the file containing the failed records.
- **Level 3 — Employee ID** (e.g. 35196, 35643): The specific employee record that failed. Expand to see the file it came from.
- **Level 4 — File Name** (e.g. IFF_BE01_202602091616): The exact source file that contained this employee's record.

Cells with a **dark red / highlighted fill** indicate a field value that has failed a DQ rule for that employee. Cells with no fill are either passing or not applicable for that field.

> 💡 **Hover Interaction — See the Exact Error**
> Hovering over any **highlighted (red) cell** reveals a tooltip with the full DQ rule failure detail, including:
> - **Which DQ Dimension failed** (e.g. "DQ Dimension Validity KO For ASSIGNMENT_CATEGORY")
> - **The rule description** — the plain-language explanation of what the field must satisfy (e.g. "Assignment category must be valid — the value input must belong to the list of authorised values for this data point")
> - **The specific error message** — the precise reason this record failed (e.g. "ERROR: These combinations of action code, worker type and assignment category are not managed")
>
> This means you do not need to cross-reference the rule registry manually — the exact failure reason is surfaced directly on the cell, making it fast to diagnose and communicate back to the source entity.

## 🎛️ Filters Panel (Right Side)
The page includes four slicers to narrow down the error log quickly:

| Filter | What it does |
|--------|-------------|
| **Reported Date** | Filter rejections to a specific day or date range within the week. |
| **Entity** | Isolate errors for a specific AXA entity (e.g. view only AXA Belgium records). |
| **Employee** | Search for a specific employee ID to check their individual rejection history. |
| **File Name** | Filter to a specific source file submission to investigate a particular interface run. |

These filters work together — combine Entity + Reported Date to quickly scope down to a single entity's problematic file run.

## 🧭 How to Navigate

- **Click "Data Collect – Interface Monitoring"** → Returns to the collection volume summary.
- **Click "DQ Monitoring"** → Goes back to the entity-level DQ score view.
- **Click "Review DQ Rules"** → Opens the full rule registry to look up the Functional ID behind any error you find here.
- **Click "Home"** → Returns to the main summary dashboard.
- **Click "Summary"** → Opens the consolidated weekly summary view.
- **Click "Support"** → Raises a support request or accesses documentation.
