## 🔍 How to Use Filters

Filters let you narrow the dashboard to a specific time period, entity, employee, or data category. Each page has its own set of filters tailored to what you can investigate there. All filters work together — applying multiple filters at once further narrows your results.

---

### 📄 Data Collect – Interface Monitoring Filters

| Filter | Options / Behaviour | What it does |
|--------|-------------------|-------------|
| **Date Range** | 3/1/2026 – 3/7/2026 (date picker) | Limits all visuals to records reported within the selected date window. Adjust both start and end dates to zoom into a specific week or day range. |
| **Entity** | Dropdown — defaults to "All" | Filters all KPIs and charts to a single AXA entity (e.g. AXA Belgium, AXA UK). Select "All" to see the consolidated view. |
| **Is Executive** | Toggle — No / Yes | Filters records by whether the employee is flagged as an executive in the source system. Use this to assess whether executive-level records have different error patterns. |
| **Log Type** | Checkboxes — FUNCTIONAL / HDL / MDM | Filters records by the system that logged the interface event. **HDL** = HCM Data Loader, **MDM** = Master Data Management, **FUNCTIONAL** = functional-level processing logs. |
| **Data Category** | Checkboxes — (Blank) / Compensation / Job / Personal | Filters records by data category. Useful for isolating collection issues to a specific HR data domain. |

---

### 📄 DQ Monitoring Filters

| Filter | Options / Behaviour | What it does |
|--------|-------------------|-------------|
| **Date Range** | 3/1/2026 – 3/7/2026 (date picker) | Scopes DQ scores and the field-level table to records processed within the selected dates. |
| **Entity** | Dropdown — defaults to "All" | Filters DQ scores, the entity bar chart, and the field-level detail table to a single AXA entity. |
| **Is Executive** | Toggle — No / Yes | Isolates DQ analysis to executive or non-executive employees, useful for understanding if quality issues are concentrated in a specific workforce segment. |
| **DQ Dimension** | Checkboxes — Completeness / Validity | Filters the field-level DQ table and score visuals to only the selected quality dimension(s). Use this to focus on one type of failure at a time. Note: Consistency may appear separately depending on active rule coverage. |
| **Data Category** | Checkboxes — (Blank) / Compensation / Job / Personal | Filters the DQ detail table and entity scores to records within the selected data category, helping you isolate whether quality issues are in Job, Personal, or Compensation fields. |

### 📄 Investigate Errors Filters

| Filter | Options / Behaviour | What it does |
|--------|-------------------|-------------|
| **Date Range** | 2/1/2026 – 2/28/2026 (date picker) | Limits the rejected rows matrix to records reported on dates within the selected range. Use this to focus on a specific day's file submission. |
| **Entity** | Dropdown — defaults to "All" | Filters the error matrix to only show rejected rows from the selected AXA entity. |
| **Is Executive** | Toggle — No / Yes | Filters rejected records to executive or non-executive employees. |
| **Functional ID** | Dropdown — defaults to "All" | Filters rejected rows to records that failed a specific DQ rule, identified by its Functional ID (e.g. DQ-VA-0001). Use this to scope the matrix to one specific rule violation across all entities. Cross-reference with the Review DQ Rules page to understand the rule logic. |
| **Data Category** | Checkboxes — (Blank) / Compensation / Job / Personal | Narrows the error matrix to rejected rows within a specific data category. Combine with Functional ID to pinpoint exactly which rule is failing for which data type. |

## ❓ General Questions

**Q: What does the dashboard cover?**
The dashboard monitors the weekly data collection and quality processing of HR employee records sent by AXA entities to the Group HR Data Factory via Oracle PaaS interfaces. It tracks what was received, what passed or failed validation, and why.

---

**Q: How often is the data refreshed?**
The dashboard refreshes on a scheduled basis. The last refresh date is displayed at the top of every page. Always check this date before drawing conclusions from the numbers shown.

---

**Q: What is the difference between Accepted Rows and Rejected Rows?**
- **Accepted Rows** passed all DQ validation rules and are available for downstream HR reporting and analytics.
- **Rejected Rows** failed one or more DQ rules and are held back. They are not included in Group HR reporting until the source entity corrects and resubmits them.

---

**Q: What do the three DQ Dimensions mean?**
- **Completeness** — A mandatory field is empty or missing. The record was submitted but key data was not included.
- **Consistency** — Field values are present but logically contradict each other (e.g. incompatible combinations of action code, worker type, and assignment category).
- **Validity** — A field value does not match the allowed list of values or the required format for that data point.

---

**Q: What is the difference between HDL and MDM Log Types?**
- **HDL (HCM Data Loader)** — Errors logged during the file loading stage into Oracle HCM. These typically relate to format or structural issues in the submitted file.
- **MDM (Master Data Management)** — Errors logged during the data validation and mapping stage. These relate to values that don't match reference data or business rules managed centrally.
- **FUNCTIONAL** — Errors raised at a higher processing level, typically related to business logic that spans multiple fields or entities.

---

**Q: What does a Functional ID mean (e.g. DQ-VA-0001)?**
The Functional ID is the unique code for a specific DQ rule. The prefix tells you the dimension:
- **DQ-VA-** = Validity rule
- **DQ-CO-** = Completeness rule
- **DQ-CS-** = Consistency rule

Go to the **Review DQ Rules** page and search for the Functional ID to read the exact rule description and understand what needs to be fixed.

---

**Q: A cell is highlighted red in the Investigate Errors page — what do I do?**
Hover over the red cell to see the exact DQ rule that failed, the dimension it belongs to, and the specific error message. Note the **Functional ID**, then go to **Review DQ Rules** to read the full rule definition. Use this information to advise the relevant entity's data team on what needs to be corrected in their source system before the next file submission.

---

**Q: Why does a field show "NA" in the DQ Score Details table?**
"NA" means that particular DQ dimension (Completeness, Consistency, or Validity) is **not evaluated** for that specific field. Not every field is checked against every dimension — the applicable rules depend on the business logic defined for that data point.

---

**Q: Why is my Overall Error Rate high even though my DQ Score looks acceptable?**
The Overall Error Rate reflects the percentage of rows with **at least one error** across any dimension, while the DQ Score reflects the average quality level across all evaluated rules. A record can have one minor Validity warning and still count toward the error rate, even if 95% of its fields are clean.

---

**Q: Can I filter across multiple entities at once?**
The Entity filter currently supports selecting one entity at a time from the dropdown. To view all entities together, select "All" from the dropdown, which is the default state.

---

**Q: What is the "Is Executive" filter used for?**
This filter separates executive-level employees from the general workforce population. It allows HR leadership and governance teams to assess whether data quality issues are evenly distributed or disproportionately affecting a specific workforce tier.
